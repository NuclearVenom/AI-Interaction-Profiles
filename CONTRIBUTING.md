# Contributing

Thank you for your interest in contributing to AI Interaction Profiles.

This repository is a documentation-first proof-of-concept for designing better human-AI interaction. Contributions should improve clarity, profile quality, examples, evaluation criteria, or documentation consistency.

## What This Project Is

This project explores whether carefully designed Interaction Profiles can improve conversations with existing large language models. It is not a software project, browser extension, chatbot, model wrapper, API, or product plan.

Please keep contributions aligned with that scope.

## Good Contributions

Useful contributions include:

- clearer explanations of the interaction problem;
- better before-and-after conversation examples;
- improvements to design principles;
- additions to the FAQ;
- refinements to existing profile pages;
- copyediting for consistency and voice;
- thoughtful critique of assumptions;
- suggestions for lightweight evaluation methods.

## Contributions To Avoid

Please do not submit:

- application code;
- browser extension designs;
- API designs;
- technical build specifications;
- exaggerated marketing claims;
- claims of scientific proof without evidence;
- platform-specific instructions that imply endorsement by a particular AI provider.

## Writing Style

Use a calm, research-inspired tone. Prefer language such as:

- "This repository explores..."
- "This proof-of-concept investigates..."
- "We hypothesize..."
- "This concept suggests..."

Avoid hype, certainty beyond the evidence, or language that makes users feel inadequate.

## Terminology

Prefer:

- Interaction Profile;
- interaction layer;
- interaction behavior;
- profile category;
- profile version.

Avoid framing the repository as a prompt collection or task instruction library. Existing personalization features may be mentioned as a usage mechanism, but the concept should come first.

## Documentation Standards

When editing documentation:

- keep claims modest and testable;
- link related documents where useful;
- define terms before relying on them;
- use realistic examples;
- preserve the distinction between current proof-of-concept work and future research directions;
- avoid turning future ideas into technical specifications;
- remove duplicated explanations when a link would be better.

## Pull Requests

Before opening a pull request:

1. Read the [Code of Conduct](CODE_OF_CONDUCT.md).
2. Check that your change fits the current scope.
3. Keep the pull request focused on one topic.
4. Explain why the change improves the repository.

For larger conceptual changes, please open an issue first.

## Review Criteria

Maintainers may ask:

- Does this improve conceptual clarity?
- Does it respect user agency?
- Does it avoid product-build detail?
- Does it avoid overstated claims?
- Does it make the project easier to understand or evaluate?
- Would the suggested interaction behavior remain useful across repeated use?
